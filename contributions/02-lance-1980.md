## Contribution Summary

This contribution focuses on Lance issue #1980: adding an `n_jobs` parameter to `lance.write_dataset` to speed up writing large in-memory tables.

The issue reports that writing many large in-memory tables one at a time underuses available CPU and network resources. The proposed improvement is to expose a parallelism parameter so users can write large in-memory data more efficiently.

For Phase I, I completed repository setup and codebase orientation. ---

## Issue Overview

The issue asks for a new `n_jobs` parameter in `lance.write_dataset`.

The reported motivation is performance. When writing large in-memory tables, the current single-call writer may not fully use available CPU and I/O resources. The issue suggests that parallelizing writes across multiple in-memory tables or chunks could reduce total write time.

The main technical question is where parallelism should live:

* at the Python API level,
* in the native Python-to-Rust bridge,
* or in the Rust writer implementation.

The current goal is not to implement immediately. The next step is to reproduce or baseline the current behavior and understand the safest implementation path.

## Phase I Findings

The issue still appears valid based on the initial audit.

Key findings:

* `lance.write_dataset` currently has many write options, but no direct `n_jobs` parameter.
* Existing options include sizing and storage-related controls, such as rows per file, rows per group, bytes per file, storage options, and dataset format options.
* Some concurrency-related options exist elsewhere in the repository, but they do not appear to expose parallel write behavior for `lance.write_dataset`.
* The Rust writer path appears to process a single stream of record batches.
* The issue likely requires careful design because parallel writes may affect fragments, commits, dataset consistency, and transaction behavior.

Current conclusion:

```text
Issue #1980 appears valid, but implementation should not begin until Phase II reproduction/baseline confirms the behavior and clarifies the safest design.
```

## Phase II Reproduction

Phase II is complete.

I wrote a temporary local benchmark script outside the committed source tree:

```text
/tmp/lance_issue_1980_repro.py
```

### What it compared

The script compared two write paths for the same in-memory table:

1. The current single-call `lance.write_dataset`.
2. An external threaded fan-out using `lance.fragment.write_fragments` plus `LanceDataset.commit`.

For each run it recorded elapsed write time, final row count, number of fragments, row-count match after read-back, and a data-content match against the source table.

### Results

These are local results from temporary-directory datasets on a single Mac, not production-scale measurements.

Run 1:

```text
rows: 50,000   dim: 32   jobs: 2
single:   0.022s   fragments: 1
parallel: 0.009s   fragments: 2
speedup:  2.49x
correctness: PASS
```

Run 2:

```text
rows: 200,000   dim: 64   jobs: 2
single:   0.049s   fragments: 1
parallel: 0.031s   fragments: 2
speedup:  1.60x
correctness: PASS
```

Run 3:

```text
rows: 200,000   dim: 64   jobs: 4
single:   0.039s   fragments: 1
parallel: 0.023s   fragments: 4
speedup:  1.72x
correctness: PASS
```

### Interpretation

* The current `lance.write_dataset` API does not expose an `n_jobs` parameter.
* The temporary external workaround produced correct datasets in all local runs: row counts and data content matched the source table.
* The parallel fan-out created multiple fragments and showed faster elapsed time than the single-call path in every local baseline run.
* These results are small, local, temp-dir measurements. They are not proof of production-scale behavior, and the speedup numbers will vary with hardware, data size, and storage backend.
* This evidence supports continuing to implementation planning. The implementation path should still be kept small and may need maintainer guidance, because write behavior touches fragments and commits.

## Risks and Open Questions

Open questions before implementation:

* Should `n_jobs` be implemented at the Python API level or in Rust `WriteParams`?
* Should `n_jobs=None` preserve current behavior exactly?
* Should invalid values like `0` or negative numbers raise an error?
* Should streaming inputs support `n_jobs`, or should the parameter only apply to in-memory tables?
* How should parallel writes preserve fragment ordering, commit behavior, and dataset correctness?
* What level of benchmark evidence is enough for maintainers?

Risk areas:

* Over-scoping a first PR.
* Changing write behavior for existing users.
* Adding Python-only behavior that maintainers may prefer to handle in Rust.
* Creating parallel writes without clear transaction semantics.
